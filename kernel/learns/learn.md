# Linux 内核开发检视指南
## 从经验到协议的 review 方法与知识组织

这份文档总结了本仓库在 Linux kernel patch review、regression analysis、false-positive elimination 与 subsystem-specific reasoning 上沉淀出的经验。

它不是一份通用代码审查建议，也不是 prompt 使用说明。它试图回答三个更具体的问题：

1. 内核代码检视真正困难的地方是什么。
2. 一次高质量 review 应该如何组织。
3. 哪些经验值得沉淀为长期可复用的规则。

本仓库的核心价值，不在于“列出更多可能的 bug”，而在于把检视过程组织成一套可执行协议：先收集上下文，再拆分变更类别，再验证可达性，再排除误报，最后以社区可接受的方式报告结论。

因此，这份指南既是方法论说明，也是经验沉淀规范。

## 1. 文档目标与适用范围

这份指南面向三类读者：

- 使用这套 review prompts 的读者
- 想把内核检视经验系统化的人
- 继续维护和扩展本仓库的人

它的目标不是替代具体 prompt，而是给这些 prompt 提供统一的设计原则和知识组织框架。

适用范围主要包括：

- Linux kernel patch review
- 回归分析与风险验证
- false positive 控制
- 子系统特定经验沉淀
- review 输出风格规范

不适用范围包括：

- 一般性的代码风格 lint 建议
- 与内核无关的通用代码审查经验
- 某个单一 patch 的一次性笔记

## 2. 内核代码检视的核心哲学

内核 review 的目标不是“提出怀疑”，而是“证明行为变化是否会引入回归”。

与普通应用代码不同，内核 patch 往往同时具备以下特点：

- 上下文分散在调用链、配置项、并发语义和子系统约束中
- 局部修改可能在未改动的 caller 或 callee 中产生副作用
- 许多看似合理的担忧，最后会被证明只是误报
- 报告的表达方式本身会影响维护者是否愿意继续读下去

因此，内核检视应当遵循以下原则。

### 2.1 不把 diff 当作分析边界

改动发生在 diff 中，但回归发生在执行路径中。只看 diff hunk，通常只能得到局部语法变化；真正需要判断的是行为变化如何沿调用栈、资源生命周期和并发边界传播。

### 2.2 不把“可能出错”当作“已经证明出错”

内核 review 中最常见的失败，不是漏报，而是误把理论上的担忧当成现实中的 bug。任何问题在进入最终报告前，都应尽量转化为可达、可触发、可解释的具体失效模式。

### 2.3 先做阶段化分析，再做局部判断

不要看到一个可疑点就立刻下结论。先完成上下文收集、变更分类和 reachability 检查，再进入 regression judgement。顺序本身就是减少误报的重要机制。

### 2.4 通用知识不足以覆盖子系统语义

锁、引用计数、RCU、VFS、block、BPF、MM 等都有自己的局部契约。经验必须按子系统分层沉淀，而不是依赖“C 语言常识”做泛化判断。

### 2.5 报告风格也是 review 质量的一部分

正确但难读、过度武断、偏离社区沟通习惯的报告，实际价值会大幅下降。高质量 review 不仅需要结论正确，也需要表达克制、证据充分、结构清楚。

## 3. 一次高质量 kernel review 的标准流程

一次完整的内核 patch 检视，至少应经过以下阶段。

### 3.1 建立上下文，而不是直接下判断

先理解补丁的意图，再理解补丁修改了哪些函数、类型、资源关系和控制流。

最常见的早期错误是：只读 commit message，或只看一两处 diff hunk，就开始判断对错。正确做法是先读完整 diff，识别被修改的函数、类型和调用关系，再决定需要补充哪些上下文。

### 3.2 把 patch 拆成更细的 change categories

一个 patch 可能同时修改：

- 控制流
- 返回值语义
- 资源管理
- 初始化逻辑
- 锁与并发边界
- 子系统 API 用法

这些变化不应混在一起讨论。每一类变化都应单独分析其风险，因为不同类别的问题需要不同的证据和不同的验证方法。

### 3.3 先做 reachability，再做 regression judgement

在分析某个风险之前，先回答：这条路径在当前工作负载、配置条件、协议约束和调用关系下是否真的能走到？

如果不能证明路径可达，就不应轻易进入最终结论。对 crash、deadlock、memory corruption 这类问题，可达性的判断应关注“结构上是否可能发生”，而不是“是否每次都会发生”。

### 3.4 把调用栈当作最小分析单位

非 trivial 变更不应只停留在“被改函数”本身。至少应检查：

- 谁调用它
- 它调用谁
- 错误路径是否仍然闭合
- 锁是否在整个作用域内有效
- 资源所有权是否跨边界传播

局部看似安全的变化，往往在 caller 对返回值的假设、callee 对锁的要求、或 cleanup path 的顺序中产生问题。

### 3.5 先假设 patch 有问题，再要求证据证明它没问题

这不是鼓励偏见，而是为了防止在复杂内核代码里过早相信注释、commit message 或局部防御逻辑。经验上，更可靠的做法是先把任何声明视为待验证陈述，再用代码路径、配置条件和子系统规则去核实。

### 3.6 在最终报告前，单独进行误报清理

误报清理不是可选补充，而是 review 的最后一段必经流程。它的目标不是让分析更保守，而是要求每个问题都带着证据站住。

这个阶段最应问的不是“这个问题看起来像不像 bug”，而是“我是否已经证明它在今天的代码里真实成立”。

### 3.7 输出前最后做一次“可读性检查”

一条正确但结构混乱的 review，效果仍然可能很差。最终输出前，应检查：

- 是否只保留了真实问题
- 是否用尽量短的段落表达
- 是否把问题放在相关 diff 旁边
- 是否避免了无意义的情绪化措辞
- 是否符合内核社区熟悉的 plain-text 沟通习惯

## 4. 知识分层模型

本仓库中的经验不应被视为同一层级的信息，而应被组织成不同职责的文档。

### 4.1 Core workflow

定义 review 的阶段、顺序、输出要求和条件分发逻辑。

这类文档回答的是：“这次 review 应该怎么推进？”

典型内容包括：

- 任务阶段
- 必做检查
- 条件加载
- 输出格式

### 4.2 Technical patterns

沉淀跨子系统重复出现的技术判断。

这类文档回答的是：“遇到某类代码模式时，通常应该怎么想？”

典型内容包括：

- RCU 顺序
- 资源生命周期
- NULL dereference 误判边界
- ERR_PTR 与 NULL 的区别
- 锁作用域而不是仅锁存在与否

### 4.3 Subsystem guides

沉淀局部语义、不变量和 API 契约。

这类文档回答的是：“在这个子系统里，什么做法是危险的，什么检查是高价值的？”

典型内容包括：

- 子系统不变量
- 特定 API 语义
- 常见失效模式
- quick checks

### 4.4 False-positive control

规定如何把“怀疑”变成“已验证问题”，以及如何淘汰站不住脚的问题。

这类文档回答的是：“什么证据才足以进入最终报告？”

### 4.5 Reporting rules

约束最后的表达方式。

这类文档回答的是：“即便结论正确，怎样写才符合社区沟通风格？”

## 5. 经验提炼方法：从一次 bug 到一条规则

不是每个 bug 都值得沉淀成知识条目。经验要进入长期文档，至少应满足以下条件之一：

- 它在多个 patch 或多个子系统中重复出现
- 它代表一种高误报率但高价值的判断边界
- 它揭示了某类 API、锁或资源模型的稳定不变量
- 它能够被压缩成简洁的 review check

一个可复用的经验条目，通常经历以下转化过程：

1. 从单个 patch 中识别一个真实问题
2. 提炼出触发该问题的最小行为模式
3. 明确这类问题依赖的前置条件
4. 明确哪些相似情况不应误报
5. 把它压缩成 invariant、failure mode 和 checks

如果一条经验无法脱离原 patch 的具体背景独立成立，它更适合停留在案例说明里，而不是进入长期知识库。

## 6. 子系统经验的写法

优秀的子系统文档不应该是“这个子系统的背景介绍”，而应该是“这个子系统中高价值的 review judgement memo”。

推荐结构如下：

### 6.1 先写不变量

先定义这个子系统里哪些条件必须成立。例如：

- 哪些锁保护哪些状态
- 哪些对象必须先从数据结构移除，再进入延迟释放
- 哪些 API 返回值对 caller 有稳定语义

### 6.2 再写失效模式

说明违反不变量后会出现什么问题，例如：

- use-after-free
- stale state
- counter mismatch
- cleanup path 漏洞
- 错误的 callback 时序

### 6.3 最后写 quick checks

quick checks 的目标不是替代完整分析，而是在 diff 初读阶段快速定位值得深挖的位置。

好的 quick check 应尽量短，并且能够直接用于 review，例如：

- 是否在修改共享状态前后维持了锁的整个作用域
- 是否在 RCU 回调前已经把对象从查找结构中摘除
- 是否使用了真正能证明数据存在的 guard，而不是语义相近但错误的 guard

### 6.4 不写教科书，不写漫长背景

子系统 guide 的价值不在于全面，而在于高密度。它应该优先记录“判断边界”和“高风险模式”，而不是做完整子系统导论。

## 7. 常见误报来源与纠偏机制

内核 review 中的误报并不随机，它们通常来自几类稳定错误。

### 7.1 把语法风险误当成语义风险

看到某个指针访问、计数器变化或 return value 改动，并不自动意味着 bug。必须结合调用关系、配置条件和生命周期去判断。

### 7.2 只看到局部防御缺失，没有证明输入真的不可信

“这里为什么不加个 NULL check” 这类直觉非常常见，但若无法证明不可信输入能到达这里，这种建议通常只是防御式编程偏好，不是 concrete bug。

### 7.3 忽略调用者已满足的前提条件

许多函数在自身内部并不显式检查所有前提，而是依赖 caller、锁上下文、配置分支或框架层约束。未检查 caller 就下结论，是误报的重要来源。

### 7.4 忽略异步清理或所有权转移

资源被放入 list、queue、callback 或 delayed work 中，并不等于泄漏。review 时应优先追踪 ownership chain，而不是只看当前函数是否立即 free。

### 7.5 用注释替代实现验证

注释和 commit message 能帮助理解意图，但不能替代代码验证。任何基于文档进行 dismiss 的判断，都应回到实际实现确认。

### 7.6 把“unlikely”错当成“impossible”

对 deadlock、infinite wait、crash、data corruption 这类问题，稀有并不等于不存在。只要结构上可能发生，就应视为真实风险；只有在代码结构上不可能到达，才能 dismiss。

## 8. 输出风格与社区沟通约束

review 的输出不是独立文学作品，而是要嵌入现有社区沟通环境。因此，表达本身应服务于可读性和可采纳性。

### 8.1 事实优先，语气克制

报告应以技术观察为中心，而不是情绪化评判。尽量把问题写成具体问题或具体验证点，而不是武断结论。

### 8.2 段落短，问题贴着代码写

不要先写一大段总结，再在最后抛一个问题。更好的方式是让每个问题尽量靠近相关 diff，使用短段落组织证据。

### 8.3 避免无意义标签和夸张措辞

大写标题、戏剧性词汇、过度强调严重性，通常不会增强说服力，反而会让输出更不像内核社区的正常交流。

### 8.4 不用 line number，优先用函数名、调用链和小片段代码

line number 往往只在本地上下文中有意义。更稳妥的定位方式是函数名、文件路径、调用链和足够短的代码片段。

### 8.5 只保留问题本身需要的上下文

如果 quote 太长，真正的问题会被淹没。应尽量 snip 无关内容，只保留让问题成立所必需的最小上下文。

## 9. 新文档准入标准

为了让知识库持续扩展而不退化，新增经验应满足以下要求。

### 9.1 一条规则必须解决一个明确判断问题

不要写成泛泛的“注意事项”。应回答具体问题，例如：

- 哪种调用顺序会引入 use-after-free
- 哪种 guard 不能证明 bio 有 data
- 哪种错误路径会丢失锁释放

### 9.2 每条规则都应尽量包含三个组成部分

- invariant
- failure mode
- review checks

### 9.3 只沉淀可复用经验，不沉淀一次性观察

如果一条结论只适用于某个 patch 的局部上下文，不适合写入长期文档。

### 9.4 优先写“判断边界”，其次写“背景知识”

最有价值的经验，往往不是解释概念本身，而是指出：

- 什么情况可以判断有问题
- 什么情况不能贸然判断
- 哪种常见直觉其实是误报来源

### 9.5 用最短文本承载最高密度判断

经验文档不是教程。应尽量压缩修辞，保留规则、条件和例外。

## 10. 维护与演进建议

这套知识体系的长期价值，不取决于文档数量，而取决于结构是否清晰、规则是否可靠。

### 10.1 优先补强高误报区域

新经验的优先级，不一定取决于 bug 是否罕见，而更应看：

- 这类问题是否容易误报
- 这类规则是否能显著提升 review 质量
- 这类经验是否跨多个 patch 反复出现

### 10.2 优先维护“分发逻辑”而不是一味堆规则

一个好的知识库不仅要有内容，还要有好的入口和条件加载逻辑。若知识越来越多，但加载逻辑混乱，实际使用效果会快速下降。

### 10.3 让子系统文档保持窄而深

与其写“大而全”的 subsystem guide，不如保持每份文档聚焦于少数高价值模式。这样更利于维护，也更利于在 review 时按需加载。

### 10.4 定期清理低价值规则

若一条规则长期只制造误报、缺少稳定证据或与现有实现已不匹配，应及时修订或删除。知识库不是只增不减的档案馆，而应是持续校正的判断系统。

## 附录 C：经验条目写作规范

新增经验条目时，建议逐项自查：

- 这条规则是否在多个 patch 中可复用
- 它是否定义了明确的判断边界
- 它是否同时写清楚了适用与不适用条件
- 它是否能压缩成短小、明确的 review checks
- 它是否避免了“教程化”背景铺陈
- 它是否能帮助减少误报，而不是只增加怀疑点

如果上述问题中多数回答是否定的，这条内容更适合作为案例说明，而不是知识库规则。
